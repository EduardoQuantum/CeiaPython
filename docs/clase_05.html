Diagnóstico, imputación y log de ediciones

1. Los Valores Ausentes en los Datos (Declaración de Principios)

Trabajar con datos reales implica lidiar con valores ausentes. Las bases de trabajo suelen contener datos ausentes o incorrectos: variables como ingreso, edad, etc. pueden estar ‘vacías’ o con datos missing. Pandas representa esos vacíos con NaN (Not a Number).

El tratamiento de los NaN escapa a nuestros objetivos, pero sí vamos a enfocarnos en las herramientas que Pandas permite para su tratamiento: diagnosticarlos, editarlos y llevar un registro de todo lo que se hizo. Todo de forma trackeable y reversible.

Este es un punto muy relevante: todo lo que hacemos en procesamiento de datos tiene que tener una ‘vuelta atrás’ y un ‘¿a ver cómo se hizo?’ Para nuestro yo del futuro y/o para el equipo con el que trabajemos.

Nuestro estilo de procesamiento sigue un principio sencillo de declarar pero muchas veces difícil de cumplir:

base cruda  →  diagnóstico  →  edición  →  log  →  base limpia

La base cruda nunca se toca. Cada edición se declara, se registra y se aplica sobre una copia, que será nuestra base de trabajo. ¿Por qué difícil de cumplir? Porque muchas veces vemos un pequeño conflicto / error / ausencia de datos que digitando sobre la fuente lo arreglamos… ¡NO LO HAGAN! ¿Por qué? Hay muchas razones, vamos a las más importantes: la base origen puede cambiar y con ello llevarse todos nuestros arreglos manuales sin documentar… todo el trabajo perdido… Otra: no confíen en la memoria “hago este arreglo/cambio porque [ponga aquí cualquier motivo válido]” una semana después no saben por qué lo hicieron. En cambio si tienen un documento de edición, solo tienen que leerlo y además si fueron tan amables con ustedes mismos como para recordarles el por qué de los cambios / arreglos… fin del problema.

2. Diagnosticar NaN en el DataFrame

El primer paso ante cualquier base es preguntar: ¿hay valores ausentes? Elegimos dos herramientas de Pandas.

2.1 Visualización general con isna()

El método .isna() recorre el DataFrame y devuelve una máscara booleana: True donde hay un NaN, False donde hay dato. Combinado con .sum(), obtenemos el panorama general.

# Bloque de datos de ejemplo. Dejamos "base_trabajo" montada en la memoria.

      'H_016', 'H_017', 'H_018', 'H_019', 'H_020'
    ],
    'region': [
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro'
    ],
    'tipo_vivienda': ['Casa','Depto','Casa','Casa','Depto',
                      'Depto','Casa','Depto','Casa','Depto',
                      'Casa','Depto','Casa','Casa','Depto',

                  44, 31, 57, 30, 42, 35, 51, 46, 34, 39]

# BASE CRUDA: el dato original nunca se modifica
base_cruda = pd.DataFrame(data).set_index('cod_hogar')

# COPIA DE TRABAJO: todo se hace sobre esta copia
base_trabajo = base_cruda.copy()

Ahora si invocamos .isna() que como es un metodo lo invocamos con un punto.

# NIVEL 1: ¿cuántos NaN hay? (valor simple)
# .isna() detecta los NaN y devuelve True/False
# .sum() suma los True (True = 1, False = 0)
print("=== NIVEL 1: Diagnóstico general ===")
print(base_trabajo.isna().sum())

# Guardamos la cantidad de NaN en ingreso
# ver el uso de llaves {}: imprime el resutado/valor de una variable
n_nan_ingreso = base_trabajo['ingreso'].isna().sum()
print(f"NaN en ingreso: {n_nan_ingreso}")

Ahora listmos los registros, usamos la misma base del bloque de datos de ejemplo

# NIVEL 2: ¿qué registros tienen NaN? (lista de índices)

# Filtramos la base para quedarnos solo con las filas
# donde ingreso es NaN
# casos_con_na: DataFrame con las filas que cumplen la condición
casos_con_na = base_trabajo[base_trabajo['ingreso'].isna()]

# Extraemos los índices (H_002, H_004, etc.) como lista
# list(): convierte los índices del DataFrame a lista de Python
# .index: propiedad del DataFrame que contiene las etiquetas de fila
lista_indices = list(casos_con_na.index)

# len() cuenta cuántos elementos tiene la lista
# Si hay al menos 1 (mayor que 0), mostramos los índices
# Esto evita imprimir una lista vacía cuando no hay NaN
if len(lista_indices) > 0:
    print("=== NIVEL 2: Registros afectados ===")
    print(f"Índices con NaN en ingreso: {lista_indices}")

Otro ejemplo para individualizar registros. Usamos dos estilos, bracket y query. Este codigo puede crecer y complejizarse con todo tipo de instrucciones anidadadas

# NIVEL 3: buscar CUALQUIER patrón (no solo NaN)
# Dos estilos de filtrado: bracket y query()
print("=== NIVEL 3: Búsqueda por patrón ===")

# ESTILO 1 — BRACKET: base_trabajo[condición][columna]
# El primer bracket [] aplica la máscara y filtra filas
# Los brackets internos ['edad_jefe'] seleccionan la columna
# Los paréntesis > 50 son del operador de comparación
print("Hogares con edad_jefe > 50:")
print(base_trabajo[base_trabajo['edad_jefe'] > 50][['edad_jefe']])

# ESTILO 2 — QUERY: base_trabajo.query('condición')[columna]
# query() es un método, por eso lleva paréntesis ()
# La condición va como texto entre comillas simples
# El bracket externo selecciona la columna a mostrar
print("Hogares en GBA o NOA:")
print(base_trabajo.query('region in ["GBA", "NOA"]')[['region']])
Lo que estamos haciendo — Los 3 niveles del diagnóstico
  1. base_cruda: Es la base original. Nunca la modificamos. Si algo sale mal, siempre podemos volver a este punto.
  2. base_trabajo = base_cruda.copy(): Creamos una copia explícita. A partir de acá, todas las ediciones van sobre base_trabajo.
  3. Nivel 1 — base_trabajo.isna().sum(): Valor simple. Cuenta los NaN por columna. Es el primer control de salud.
  4. Nivel 2 — list(casos_con_na.index): Lista de índices. Nos dice qué registros revisar, no solo cuántos. La lista se prepara antes del if para que esté disponible siempre, y el if solo decide si se imprime o no.
  5. Nivel 3 — Diagnóstico universal: Dos estilos de filtrado. Bracket: base_trabajo[condición] usa corchetes, la condición va con paréntesis y operadores. query(): la condición va como texto entre comillas, los paréntesis son del método. Ambos usan brackets [['columna']] para mostrar la columna.
Mesa de Ayuda: isna() vs isnull()

isna() e isnull() hacen exactamente lo mismo. isna() es más corto. Elegí uno y mantenelo consistente en tus scripts.

3. El Log de Ediciones

Cuando editamos datos, necesitamos un registro de lo que hicimos. Un archivo de texto (log.txt) es la herramienta más simple y universal: lo puede abrir cualquier editor, queda como evidencia del proceso y se puede consultar después.

Cada edición debe dejar una entrada en el log con: - Qué variable se modificó - Qué tipo de edición se aplicó - Cuántos registros fueron afectados

# log_1.txt: guardamos los casos detectados
# Ruta: data/output/log_1.txt
# La carpeta data/output/ debe existir ANTES de ejecutar
# Si no existe, Python lanza "FileNotFoundError"
# with: abre el archivo y lo cierra automáticamente al terminar
# open("data/output/log_1.txt", "w"): ruta relativa, modo escritura
# "w" de write: crea el archivo (o lo sobreescribe si ya existe)
# as archivo_log: así llamamos al archivo dentro del bloque
with open("data/output/log_1.txt", "w") as archivo_log:
    # str() convierte la lista de índices a texto
    # str(lista_indices) produce algo como "['H_002', 'H_004', ...]"
    # archivo_log.write() escribe ese texto en el archivo
    archivo_log.write(str(lista_indices))
Sobre las rutas de archivos

"data/output/log_1.txt" es una ruta relativa: Python busca la carpeta data/output/ partiendo desde donde está el script.

  • Si la carpeta data/output/ no existe, Python lanza FileNotFoundError.

Troubleshooting: Si ves un error como [Errno 2] No such file or directory: 'data/output/log_1.txt', la carpeta data/output/ no existe. Creala manualmente.

Lo que estamos haciendo — Log y ediciones
  1. data/output/log_1.txt: Guardamos los casos detectados. El archivo es sintético: solo los índices. Es un registro de “qué revisar”. La carpeta data/output/ debe existir antes de ejecutar (ver callout sobre rutas).
  2. data/output/imputacion.txt: Archivo auxiliar con las instrucciones .loc de imputación. Se escribe “a mano” tras revisar el log. El pipeline lo inyecta con el Inclusor (ver sección 3.1).
  3. with open(...) as archivo_log:: Abre el archivo de texto. El bloque with lo cierra automáticamente al terminar.
  4. str(lista_indices): Convierte la lista de índices a texto plano. archivo_log.write(str(lista_indices)) escribe todo de una vez, sin loops ni cabeceras. El archivo queda sintético: ['H_002', 'H_004', 'H_010', 'H_013'].
  5. fillna(0): Imputación masiva. Reemplaza todos los NaN de golpe. Creamos ingreso_sin_na (columna nueva) para no perder el original.
  6. .loc[índice, columna] = valor: Imputación caso por caso. Apunta a una celda específica y le asigna un valor. Se usa cuando tenemos el dato real. Estas líneas NO van hardcodeadas en el script: van en imputacion.txt, que se inyecta con include().
  7. Por qué separar log de ediciones: El log es evidencia (se lee sin ejecutar código). El archivo de imputación es acción (se inyecta con include()). Separados, cualquier persona puede revisar el log y decidir si ejecutar la imputación o no.
La regla del pipeline: nunca pisar datos originales

En nuestro estilo de procesamiento, la base cruda nunca se modifica. Cada edición se aplica sobre una copia o sobre una columna nueva.

  • fillna(0) sobre base_trabajo['ingreso'] pisa el dato original.
  • fillna(0) sobre base_trabajo['ingreso_sin_na'] (una columna nueva) preserva el original.

Esta distinción parece pequeña, pero es la diferencia entre un pipeline confiable y un script que destruye información. Si después necesitamos comparar el ingreso original con el imputado, podemos hacerlo porque ambas columnas conviven en el mismo DataFrame.

3.1 El Inclusor — Inyectar código desde archivos externos

Este nombre horrible (inclusor / inyectar) no es otra cosa que una solucón para un problema: mi código el larguisimo y tiene diferentes secciones ¿tiene que estar todo junto? !es ilegible! usemos en forma metódica e inteligente diferntes “fragmentos” de codigo: pueden ser secciones, pueden ser bloques, etc etc.

En el procesamiento de datos, el pipeline suele involucrar muchas líneas de código. Si ponemos todo en un solo script, este crece y se vuelve difícil de leer y auditar. Lo ideal es separar la lógica en archivos independientes y luego “incluirlos” donde corresponda.

Creamos una función simple que lea un archivo y lo ejecute en el mismo “namespace” del script.

De paso hacemos una función.

# === HERRAMIENTAS DEL PIPELINE ===
# Definimos estas funciones UNA VEZ al inicio del script
# include(archivo): inyecta código desde un archivo externo
# log(archivo, contenido): guarda contenido en un archivo de texto
# Ambas encapsulan operaciones sobre archivos que usamos todo el tiempo

def include(archivo):
    with open(archivo, "r") as f:
        codigo = f.read()
    exec(codigo, globals())

def log(archivo, contenido):
    with open(archivo, "w") as f:
        f.write(contenido)
Namespaces y las dos herramientas del pipeline

¿Qué es un namespace?

El namespace es el “espacio de nombres” donde Python guarda las variables. Cuando hacemos base_trabajo = base_cruda.copy(), esa variable vive en el namespace de nuestro script.

include() necesita exec(codigo, globals()) para pasarle al archivo inyectado el espacio de nombres correcto. globals() le pasa “la mochila de variables” al archivo ejecutado. Sin globals(), exec() arranca en un namespace vacío y el archivo no ve base_trabajo, ni pd, ni nada.

Las dos herramientas

Herramienta Dirección Qué hace
log(archivo, contenido) escribe datos → archivo Guarda contenido en un archivo de texto
include(archivo) lee código ← archivo Inyecta instrucciones desde un archivo externo

log() encapsula el with open(...) as archivo_log: que vimos en la Sección 3: abrir un archivo y escribir en él. include() es su contraparte: en lugar de escribir, lee y ejecuta. Donde log() guarda evidencia, include() ejecuta correcciones.

Entre las dos cubrimos el ciclo completo del pipeline:

  1. Diagnosticamos el problema
  2. Guardamos el log con log()
  3. Escribimos las correcciones en imputacion.txt
  4. Las inyectamos con include()
El Inclusor — Organizar el pipeline con archivos externos

¿Por qué un Inclusor?

El procesamiento de datos suele involucrar muchas instrucciones. Si ponemos todo en un solo archivo, este crece y se vuelve difícil de leer.

El Inclusor resuelve esto: una función que lee un archivo .txt y lo ejecuta como si estuviera escrito en ese punto del script.

¿Cómo se usa?

  1. Se define include() una vez al inicio del script principal.
  2. Se escribe un archivo auxiliar (ej: imputacion.txt) con las instrucciones relevantes (líneas .loc).
  3. En el script principal se invoca: include("data/output/imputacion.txt").
  4. Python lo lee, lo ejecuta y sigue con la siguiente línea del script.
  5. Entonces no importa que “imputaciones.txt” tenga cientos de lineas!!
  6. Podemo tener todos los archivos “include” que querramos

¿Es 100% transparente?

Sí, para todo lo que hacemos. El código inyectado ve todas las variables del script: base_trabajo, base_cruda, los imports, todo. Y cualquier modificación que haga (crear columnas, asignar valores) queda disponible después de la inyección.

Lo que no funciona en un archivo inyectado son casos avanzados que no usaremos: imports relativos (from . import algo), referencias a __file__, y la numeración de líneas en mensajes de error (Python reporta la línea del exec, no del archivo). Fuera de eso, es completamente transparente. Esto es una desventaja… pero la manejarmos con sentido practico: si tenemos un erro que no podemos identificar siempre podemos “pegar” el contenido en el cuerpo principal, lanzar el scritp, detectar y arreglar el error.

Ventajas de este enfoque

  1. Organización: cada archivo tiene una responsabilidad única (log, imputación, etc.).
  2. Reutilización: el mismo archivo se puede inyectar en varios scripts.
  3. Lectura: el script principal se lee de arriba a abajo sin interrupciones.
  4. Auditoría: el archivo inyectado queda como evidencia del procesamiento, igual que el log.

De ahora en adelante

Este es nuestro estándar. En cada pipeline vamos a definir include() al inicio y vamos a separar la lógica en archivos externos.

En nuestro caso, el archivo a inyectar es imputacion.txt: contiene las instrucciones .loc que corrigen cada caso detectado en el diagnóstico.

# --- data/output/imputacion.txt ---
# Escribir este archivo a mano tras revisar el log_1.txt
# Cada línea es una instrucción .loc que corrige un caso específico
# El pipeline lo inyecta con: include("data/output/imputacion.txt")
# El código corre en el mismo namespace: ve base_trabajo sin problemas

base_trabajo.loc['H_002', 'ingreso_sin_na'] = 420000
base_trabajo.loc['H_004', 'ingreso_sin_na'] = 380000
base_trabajo.loc['H_010', 'ingreso_sin_na'] = 410000
base_trabajo.loc['H_013', 'ingreso_sin_na'] = 350000

4. Pipeline Completo: Diagnóstico + Edición + Log + Exportación

Ahora armamos el programa completo. Entra una base cruda, se diagnostica, se edita, se registra todo en un log y se exportan los resultados.

El objetivo es construir un script completo autoexplicado para cualquier persona y que pueda ejecutarse y obtener el mismo resultado. Eso es un pipeline reproducible.

# --- PIPELINE COMPLETO: EDICIÓN DE VALORES AUSENTES ---
# Integra diagnóstico, log e imputación en un solo script
import pandas as pd
import numpy as np

# === HERRAMIENTAS DEL PIPELINE: definimos UNA VEZ al inicio ===
# include(archivo): inyecta código desde un archivo externo
# log(archivo, contenido): guarda contenido en un archivo de texto
def include(archivo):
    with open(archivo, "r") as f:
        codigo = f.read()
    exec(codigo, globals())

def log(archivo, contenido):
    with open(archivo, "w") as f:
        f.write(contenido)

# 0. DATOS CRUDOS (simulacro EPH con NaN forzados)
data = {
    'cod_hogar': [
      'H_001', 'H_002', 'H_003', 'H_004', 'H_005',
      'H_006', 'H_007', 'H_008', 'H_009', 'H_010',
      'H_011', 'H_012', 'H_013', 'H_014', 'H_015',
      'H_016', 'H_017', 'H_018', 'H_019', 'H_020'
    ],
    'region': [
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro',
      'GBA', 'NOA', 'Patagonia', 'Cuyo', 'Centro'
    ],
    'tipo_vivienda': ['Casa','Depto','Casa','Casa','Depto',
                      'Depto','Casa','Depto','Casa','Depto',
                      'Casa','Depto','Casa','Casa','Depto',
                      'Depto','Casa','Depto','Casa','Depto'],
    'ingreso': [450000, np.nan, 580000, np.nan, 420000,
                490000, 310000, 620000, 280000, np.nan,
                460000, 335000, np.nan, 315000, 425000,
                500000, 325000, 630000, 295000, 415000],
    'edad_jefe': [45, 32, 58, 29, 41, 36, 52, 47, 33, 40,
                  44, 31, 57, 30, 42, 35, 51, 46, 34, 39]
}

# BASE CRUDA: el dato original nunca se modifica
base_cruda = pd.DataFrame(data).set_index('cod_hogar')

# COPIA DE TRABAJO: todo se hace sobre esta copia
base_trabajo = base_cruda.copy()

# 1. DIAGNÓSTICO: detectamos los NaN y guardamos los índices
# .isna() detecta los NaN, .sum() los cuenta
n_nan_ingreso = base_trabajo['ingreso'].isna().sum()
print(f"NaN en ingreso: {n_nan_ingreso}")

# Filtramos la base para quedarnos solo con las filas con NaN
casos_con_na = base_trabajo[base_trabajo['ingreso'].isna()]
# Extraemos los índices como lista
lista_indices = list(casos_con_na.index)
print(f"Índices afectados: {lista_indices}")

# 2. LOG: guardamos los casos detectados en un archivo
# log() es la función que definimos al inicio
# str(lista_indices) convierte la lista de índices a texto
# log() abre el archivo, escribe y lo cierra automáticamente
log("data/output/log_1.txt", str(lista_indices))

# 3. IMPUTACIÓN MASIVA: reemplazamos todos los NaN con 0
# Creamos una columna NUEVA para no perder la original

base_trabajo['ingreso_sin_na'] = base_trabajo['ingreso'].fillna(0)

# 4. IMPUTACIÓN CASO POR CASO: inyectamos las instrucciones desde imputacion.txt
# El archivo contiene las líneas .loc que corrigen cada caso
# include() lee el archivo y ejecuta el código en este mismo namespace
# base_trabajo está disponible automáticamente

include("data/output/imputacion.txt")

# 5. EXPORTACIÓN: base editada a Excel

base_trabajo.to_excel("data/output/base_editada.xlsx")
print("Pipeline completado. Base exportada: base_editada.xlsx")

Análisis de lo actuado:

Este es un abordaje de procesamiento de datos: tomamos una base, identificamos un problema (en este caso los datos NaN), diseñamos un abordaje reproducible y auditable, e integramos el resultado en una nueva versión de la base de datos.

Trabajamos con carpetas auxiliares ("data/output/") y con archivos de log y edición ("log_1.txt", "imputacion.txt").

El pipeline completo, paso a paso
  1. Base cruda (base_cruda): El dato original, inmutable. Si algo sale mal, siempre podemos volver acá.
  2. Copia de trabajo (base_trabajo = base_cruda.copy()): Todo el procesamiento ocurre sobre este objeto.
  3. Diagnóstico: isna().sum() cuenta los NaN, list(casos_con_na.index) identifica los registros.
  4. Log (log("data/output/log_1.txt", str(lista_indices))): Guarda los índices de los casos a revisar usando la función log(). Sintético, sin cabeceras. La carpeta data/output/ debe existir.
  5. Imputación masiva (fillna(0)): Reemplaza todos los NaN con 0 en una columna nueva.
  6. Imputación caso por caso (include("data/output/imputacion.txt")): Inyecta desde imputacion.txt las instrucciones .loc que corrigen cada caso puntual. El archivo se escribe a mano tras revisar el log.
  7. Exportación (to_excel): Guarda la base editada. La columna original ingreso sigue intacta para comparación.
Mesa de Ayuda: El log no se crea o la ruta no existe

Si el archivo no se generó, revisá la explicación sobre rutas en la Sección 3 (el callout-note después del log). La carpeta data/output/ debe existir antes de ejecutar. Si no existe, creala manualmente o agregá os.makedirs("data/output/", exist_ok=True) al inicio del script.

5. Trabajar con Datos Limpios

Una vez que tenemos la columna ingreso_sin_na sin valores ausentes, podemos aplicar todo lo que aprendimos en las clases anteriores sin sorpresas. La pivot_table ya no encuentra NaN y los cálculos son estables.

# --- TRABAJAR CON DATOS LIMPIOS ---
import pandas as pd
import numpy as np

# Cargamos la base editada (como si viniéramos de otro script)
ruta_entrada = "data/output/"
base_limpia = pd.read_excel(ruta_entrada + "base_editada.xlsx", index_col='cod_hogar')

# pivot_table con datos limpios
resumen = base_limpia.pivot_table(
    index='tipo_vivienda',
    columns='region',
    values='ingreso_sin_na',
    aggfunc='mean'
).round(2)

print("\nResumen de ingresos (datos imputados):")
print(resumen)
Base editada como nuevo punto de partida

La base exportada (base_editada.xlsx) contiene todas las columnas originales más las columnas nuevas creadas durante la edición. Esto significa que otro script puede cargar esta base y continuar el análisis sin necesidad de repetir la imputación. El log queda como registro de cómo se llegó a ese estado.

6. Resumen de Herramientas Utilizadas

Comando / Método Función Principal
df.isna() Devuelve una máscara booleana con True donde hay NaN.
df.isna().sum() Cuenta los NaN por columna.
df[df[‘col’].isna()] Filtra las filas donde una columna tiene NaN.
df[‘col’].fillna(valor) Reemplaza NaN por un valor (devuelve una copia).
df.copy() Crea una copia explícita del DataFrame.
with open(…) as archivo_log: Abre un archivo de texto para escritura segura.
log(archivo, contenido) Guarda contenido en un archivo de texto. Encapsula with open() + write().
if condición: Estructura de control para diagnosticar y verificar.
include(archivo) — exec(codigo, globals()) Inyecta código desde un archivo externo. Emula el #include de C.
def include(…) + def log(…) Las dos herramientas del pipeline. Una inyecta, la otra guarda.
os.makedirs(…, exist_ok=True) Crea la carpeta de salida si no existe (evita errores de ruta).
